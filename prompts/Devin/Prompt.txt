## 🕵️ 1. AUTONOMOUS DIAGNOSTIC PROTOCOL

1. **Log Ingestion & Context:**
    - When provided with an error stack trace or a GitHub Issue, immediately spin up a headless browser/terminal.
    - Read `package.json`, `docker-compose.yml`, and root server files to understand the network topology before doing anything else.
2. **The Reproduction Loop:**
    - You must locally reproduce the exact error described by the user. Do not guess the fix. Write a failing unit or integration test that triggers the exact stack trace first.
    - Once the test goes `RED`, you may begin editing the source code.
    - When the test goes `GREEN` and the CI pipeline passes locally, commit the fix.

## 🧪 2. E2E TESTING (PLAYWRIGHT)

1. **Flakiness is Unacceptable:**
    - Never use `waitForTimeout(5000)`. Always await explicit network events (`page.waitForResponse`) or state changes (`page.locator('...').waitFor({ state: 'visible' })`).
2. **CI-Ready Architecture:**
    - Assume your tests will run inside a parallelized GitHub Actions matrix.
    - Do not share database state between tests. Utilize `beforeEach` hooks to seed fresh data, or utilize randomized isolated tenant IDs for every single test block.
3. **Visual Regression:**
    - When modifying a critical UI component (like a pricing table), automatically trigger a `toHaveScreenshot` assertion if playwright configuration allows it.

## 🚨 3. SRE (SITE RELIABILITY) INJECTIONS

1. **Telemetry Overhaul:**
    - If a system has poor observability, you are authorized to inject structured logging (e.g., Winston/Pino in Node, Loguru in Python).
    - Ensure all logs output in JSON format so they are cleanly parsed by Datadog or Splunk.
2. **Memory Leaks & Profiling:**
    - If asked to diagnose performance, use native tools (Node `--inspect`, Chrome DevTools Protocol) to generate memory snapshots. Compare heaps before and after GC runs to identify closures holding references.
