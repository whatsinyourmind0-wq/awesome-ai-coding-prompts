## 📱 1. MOBILE ARCHITECTURE (EXPO & REACT NAVIGATION v7)

1. **Navigation Strictness:**
    - When asked to implement a new feature, first `grep` through the routing files (`app/` for Expo Router, or `src/navigation` for bare React Navigation).
    - All routes must be strictly typed via appropriate ParamLists. Never pass deep nested objects through route params; pass an ID and fetch the data on the destination screen instead.
2. **Global State & Storage:**
    - Explicitly check whether we use Zustand, Redux Toolkit, or Context API by reading `package.json` before writing the implementation file.
    - DO NOT use generic AsyncStorage for critical user tokens. Scan for `react-native-mmkv` or Expo SecureStore. Fallbacks and `try/catch` wrappers are non-negotiable.

## 🐎 2. PERFORMANCE & ANIMATIONS 

1. **Reanimated 3 Superiority:**
    - Use `react-native-reanimated` for everything dynamic. The legacy `Animated` API from React Native core is forbidden unless specifically requested.
    - Execute animations on the UI thread using `runOnUI` or UI worklets to ensure 120hz frame rates.
    - When modifying layout, use `LinearTransition` or `FadeIn/Out` intelligently.
2. **Diff Management Strategy:**
    - Keep diffs completely isolated to components actively being changed. 
    - Do not refactor surrounding untouched code, especially near gesture handlers (`GestureDetector` logic is extremely brittle to unexpected refactors).

## 🛡️ 3. TYPESCRIPT & RELIABILITY

1. **Interface Boundaries:**
    - Do not break existing TypeScript UI Component props. 
    - If you extend a base prop system (like upgrading a basic Button to an IconButton), use `Omit<Props, 'icon'> & { newIcon: Element }` instead of redeclaring the type completely to save diff space and mental overhead.
2. **Offline-First Resilience:**
    - All network requests (React Query / SWR / Fetch) must gracefully handle `network_error` states without crashing deeply nested components. Provide intuitive UI `<OfflineBanner>` wrappers if the context permits.
