## 🏗️ 1. MONOREPO & BUILD ARCHITECTURE

1. **Workspace Topography:** Understand the boundaries of `apps/` (consumer applications) and `packages/` (shared logic, UI, configs).
2. **Dependency Management (pnpm workspace):**
    - Never duplicate dependencies across apps if they belong in a shared package.
    - When adding a tool (e.g., Day.js, Zod), install it at the package level and export it through `packages/utils/src/index.ts`.
3. **Turborepo Configurations:** Ensure `turbo.json` aggressively caches `lint`, `test`, and `build` outputs. Inputs must be strictly defined (e.g., `^build` meaning depends on topology dependencies).

## 🧩 2. COMPONENT DESIGN (VUE 3 & COMPOSITION API)

1. **Strict `<script setup>`:** All new Vue components must use the `<script setup lang="ts">` paradigm. Options API is strictly legacy and forbidden for new code.
2. **Reactivity Mastery:** 
    - Understand the deep difference between `ref` and `reactive`. Default to `ref` for primitives and reassigned objects.
    - Avoid deep watching (`{ deep: true }`) large reactive trees unless absolutely necessary for performance reasons.
3. **Composables over Mixins:** Extract all complex business logic into pure, testable composables (`useAuth.ts`, `useBilling.ts`) in the `composables/` directory.

## 🛡️ 3. ERROR HANDLING & STATE

1. **Nuxt 3 Nitro Layer:**
    - All backend `/server/api/` routes must return strictly typed JSON structures.
    - Wrap potential failure points in `try/catch` blocks and use `createError({ statusCode: 500, message: '...' })`.
2. **Global State (Pinia):**
    - Do not over-pollute the global store in Pinia. If state only belongs to a specific page or wizard process, keep it local via Provide/Inject or a local composable.
    - Hydration mismatches are fatal. Ensure `useAsyncData` or `useFetch` are used correctly to match server HTML with client hydration.

## ✍️ 4. CODE STYLE & DIFFS

1. **Atomic File Generation:** When scaffolding a full feature (e.g., "Add Stripe Billing"), generate the Database Migration, the Server API route, the Pinia Store definition, and the UI Component in logical sequence. Do not dump them all at once if the user asks you to explain it first.
2. **Context Retention:** Trae Builder relies on the files open in the IDE. Before writing, summarize the currently open files to explicitly prove you understand the context of the requested change.
